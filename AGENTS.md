## 改善アプローチ

戦略改善では、やみくもにロジックや閾値を試さないこと。
必ず、先に「勝てない日・週の原因分析」を行い、その分析結果に対応する仮説ベースの改善だけを試すこと。
また、バックテスト期間にだけ都合よく合うカーブフィッティングを避け、将来の本番運用でも説明可能で機能しうる頑健なロジックを優先すること。

### 未知の相場への考え方

将来の相場で起こりうる全パターンを、事前に細かい条件分岐ですべて列挙して対応しようとしないこと。
そのような設計は、過去データには強く見えても、少し形の違う将来局面で壊れやすく、カーブフィッティングを招きやすい。

目指すべきなのは「全パターン対応」ではなく、「未知のパターンが来ても大損しにくい shared strategy」にすることである。
そのため、次を優先すること。

1. breadth、gap、RS、指数地合い、候補分布などから regime の違和感を捉えること
2. 自信の低い regime では、entry を無理に増やすのではなく、size を落とす、probe 化する、または no-trade を許容すること
3. 未知の regime を見つけたときは、その regime 専用の細かい例外分岐を足す前に、まず shared な leverage / notional / exit / weekly de-risk で損失を抑えられないかを検討すること
4. 採用判断では、「未知の局面でも勝てること」を求めるのではなく、「未知の局面でも壊れにくく、負けても浅く済むこと」を重視すること
5. `train` に十分な再現例がないパターンに対しては、直近 holdout の見た目だけで細かく当て込まず、一般化しやすい安全装置だけを優先すること

### Train / Holdout 運用

戦略改善では、原則として最新データの直近1ヶ月を `holdout`、それ以前を `train` として扱うこと。

1. `train` だけを使って未達週、負け日、ノートレード日を分析し、改善案を作ること
2. 閾値調整、候補比較、採用候補の順位付けは `train` だけで行うこと
3. `holdout` は採用候補の最終確認にだけ使い、近い閾値を何度も `holdout` で見比べて当て込みを起こさないこと
4. 最新データが取得できるときは、先にキャッシュを更新し、その最新日を基準に `holdout` を切り直すこと
5. 最新更新に失敗した場合は、現在のキャッシュ最新日を明示したうえで、その時点の `holdout` 定義を固定して検証を続けること
6. rolling / walk-forward 確認を行う場合も、目的は「採用候補の頑健性確認」であり、その結果を見ながら同一セッションで細かい当て込みを繰り返さないこと

最低限、次の順で進めること。

1. `jp_backtest.py` や補助集計から、未達週・負け日・ノートレード日を洗い出す
2. それらを setup、曜日、breadth、gap、RSI、RS、指数地合い、保有数、損失集中日などで分解する
3. どの条件で期待値が崩れているのか、どの条件なら未達週を埋められそうか、原因仮説を言語化する
4. その仮説に対応する shared strategy change だけを試す
5. 改善が出なかった場合は、何が仮説と違ったのかを探索ログへ残す
6. 1つの策を試して終わらず、その結果から新たに見えた問題点を再分析し、明確な次の対策がある限り続けて対応する
7. 報告は、他にやるべき明確な分析タスクや改善タスクがなくなった時点、または制約やトレードオフ上これ以上進める根拠が弱くなった時点で行う

禁止事項:

- 根拠のない広いグリッド探索を先に回すこと
- 負けの原因と無関係な閾値を片っ端から触ること
- 特定期間の数週・数日だけを都合よく改善するために、一般化しにくい細かすぎる条件分岐や閾値調整を積み重ねること
- 「総リターンが上がったから」という理由だけで、未達週や最大日次損失の悪化を無視して採用すること
- 1つの施策の検証結果だけを報告して打ち切り、そこから自然に続く再分析や次の対策を残したまま終えること

優先事項:

- まず未達週をどう埋めるか
- 次に負け日の損失集中をどう減らすか
- そのうえで総リターン、PF、稼働率のバランスを見ること

## デイトレード戦略の目標

このリポジトリの主戦略は、日本株のデイトレードである。売買ロジックは当日中に建玉を解消し、翌日にポジションを持ち越さない前提で設計すること。戦略改善では、次の目標を同時に満たす方向を優先すること。

1. 毎週、週初資産比で +1% の利益を目指すこと。
2. 月間営業日の 3/4 以上でトレード機会を持てる稼働率を目指すこと。
3. 利益最大化を狙う場合でも、現実の執行、流動性、信用余力、スリッページ、急落時損失を踏まえたロジックにすること。
4. 攻めた資金管理は許容するが、バックテスト上の利益だけを理由に、実運用で破綻しやすい過大レバレッジ、過大ポジション、極端な月次ドローダウン許容を無批判に採用しないこと。
5. 週次 +1% は保証ではなく最適化目標として扱うこと。達成率、未達週、最大日次損失、月次稼働率、プロフィットファクターを合わせて評価すること。
6. 月 3/4 以上のトレード頻度を上げるために、低品質なエントリーを無理に増やさないこと。稼働率と期待値のバランスを優先すること。
7. 戦略変更後は、`jp_backtest.py` の結果として少なくとも総リターン、取引回数、勝率、プロフィットファクター、月間稼働率、`MONTHS >= 3/4 ACTIVE`、`WEEKS >= +1%`、最大日次損失を確認すること。
8. 高利益化のための探索は行ってよいが、採用する値は本番で説明可能な共有戦略ロジックとして表現できるものに限ること。
9. 採用判断では、過去データへの当て込みではなく、将来の本番運用でも再現しうる頑健性、説明可能性、執行可能性を優先すること。

ルール:

1. 売買判断の唯一の判断源は本番戦略ロジックとすること。
2. `backtest.py`、`jp_backtest.py`、および関連テストは、本番戦略ロジックを参照して動作させること。独自の戦略判断を持たせないこと。
3. 本番や `core` 配下の戦略モジュールに、バックテスト専用ロジック、バックテスト専用設定、テスト専用の判断分岐を追加しないこと。
4. 本番とバックテストの差分として許容されるのは、実行手段の差だけとすること。具体的には、データ取得タイミング、板情報、特別気配対応、監視ループ、実注文、約定処理などの実運用依存部分に限ること。
5. 何を買うか、何を売るか、何を保有継続するか、どう資金配分するか、という戦略判断は必ず共通化すること。
6. 差分を見つけた場合は、本番ロジックにテスト用の分岐を足すのではなく、バックテスト側を本番戦略参照へ寄せることを優先すること。
7. テストやバックテストのために、本番戦略の挙動を変える feature flag や専用スイッチを追加しないこと。
8. 戦略変更を行う場合は、まず共有された本番戦略ロジックを修正し、その結果をバックテスト側から参照する形にすること。

## 推奨アーキテクチャ

役割は次のように分離する:

- 共有戦略エンジン: エントリー、イグジット、保有継続、ランキング、資金配分を決める
- 本番実行レイヤー: ライブデータ、監視、注文、約定処理を担当する
- バックテスト実行レイヤー: 共有戦略エンジンを使って仮想約定を行う

## レビュー基準

戦略コードを変更するときは、本番とバックテストの乖離を減らす設計を優先すること。
バックテスト側に独自の戦略判断が増える変更は、実行シミュレーション上どうしても必要な場合を除き、設計上の問題として扱うこと。
